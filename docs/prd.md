# 📄 제품 요구 사항 문서 (PRD) - 개발자 생산성 대시보드 (D-Prod)

## 1. 프로젝트 개요 및 목표

| 핵심 구성 요소                    | 설명                                                                                                                                         |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------- |
| **프로젝트 제목**                 | **개발자 생산성 대시보드 (D-Prod)**                                                                                                          |
| **핵심 목표 (One-Sentence Goal)** | 개발자가 다양한 플랫폼에 흩어진 업무·학습·활동 정보를 하나의 대시보드에서 직관적으로 확인하고, 자신에게 맞게 구성할 수 있도록 하는 것입니다. |
| **대상 사용자 (Target Users)**    | GitHub, Stack Overflow, 기술 뉴스, 일정 등을 자주 확인하며 생산성을 높이고 싶은 모든 소프트웨어 개발자(주니어~시니어).                       |

## 2. 핵심 기능 (MVP 위젯)

MVP에는 다음 네 가지 핵심 위젯이 포함되어야 하며, 이 모든 위젯은 **드래그 앤 드롭** 및 **기본 커스터마이징**을 지원해야 합니다.

| 위젯 이름                       | 주요 표시 정보                                                                                                                                                  | 핵심 가치 제안                                                                                      |
| :------------------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- |
| **1. GitHub 활동 위젯**         | 개인 활동 요약 (오늘/최근 7일 커밋 수, 생성한 PR 목록, 할당된 Issue 목록). **선택한 Repository 기준으로 필터링 가능해야 합니다.**                               | 개발자가 대시보드를 떠나지 않고도 일일 진행 상황 및 기여 상태를 빠르게 모니터링할 수 있도록 합니다. |
| **2. 오늘 할 일 위젯**          | 오늘 해야 할 일 목록 및 완료/미완료 상태를 위한 체크박스. 간단한 추가/삭제 기능을 지원합니다. **할 일 상태는 Local Storage를 사용하여 영구 저장되어야 합니다.** | 대시보드를 "개발자의 하루 시작점"으로 만들고, 집중해야 할 작업을 한눈에 확인할 수 있게 합니다.      |
| **3. 기술 뉴스 헤드라인 위젯**  | 사용자가 정의한 키워드를 기반으로 한 최신 기술 뉴스 헤드라인 5~10개. 출처를 표시합니다. 간단한 제목 + 링크 형식으로 구성됩니다.                                 | 개발자가 최신 기술 동향을 빠르게 파악하고 학습 흐름을 유지하는 데 도움을 줍니다.                    |
| **4. Stack Overflow 질문 위젯** | 사용자가 정의한 관심 태그(예: React, JS)와 관련된 최신 질문 5개 목록. 질문 제목, 점수(score), 답변 수(answer count)를 표시하며, 클릭 시 SO 페이지로 이동합니다. | 활발한 커뮤니티 토론 및 개발 관련 잠재적 문제에 대한 빠른 통찰력을 제공합니다.                      |

## 4. 데이터 및 영속성 요구 사항

MVP의 모든 데이터 영속성은 `Zustand persist` 미들웨어를 통해 **브라우저의 Local Storage**를 사용하여 클라이언트 측에서 처리됩니다.

| 데이터 유형                          | 저장 위치                            | 이유                                                                                               |
| :----------------------------------- | :----------------------------------- | :------------------------------------------------------------------------------------------------- |
| **A. 레이아웃 정보**                 | Local Storage (Zustand persist 활용) | 세션 간에 사용자가 커스터마이징한 드래그 앤 드롭 그리드 구조를 유지하는 데 필수적입니다.           |
| **B. 오늘 할 일 목록**               | Local Storage (Zustand persist 활용) | 오프라인에서도 유지되어야 하는 간단하고 소규모의 개인 텍스트 데이터입니다.                         |
| **C. 사용자 설정** (태그, 키워드 등) | Local Storage (Zustand persist 활용) | 외부 API에서 **어떤 데이터를 가져올지** 결정하는 설정 데이터(예: `techNewsKeywords`)를 저장합니다. |

## 5. API 통합 요구 사항

대시보드는 동적 데이터를 위해 외부 API에 의존합니다. 프론트엔드 구현 시 인증, 요청 한도 및 데이터 가져오기 복잡성을 관리해야 합니다.

| 위젯                    | 사용 API                        | 인증/통합 계획                                                                                           | 프론트엔드 기술 고려 사항                                                                                                                                                                                                                  |
| :---------------------- | :------------------------------ | :------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **GitHub 활동**         | **GitHub API** (REST + GraphQL) | 개인 활동 조회를 위해 **OAuth 기반** 인증이 필요하며, 사용자에게 필요한 권한(Scope)을 부여받아야 합니다. | **요청 한도(Rate Limit) 처리 및 캐싱 계층:** 요청을 최소화하고 한도를 준수하기 위해 클라이언트 측 캐싱 메커니즘을 구현해야 합니다. **GraphQL**을 사용하여 여러 데이터 유형(PR, 커밋)을 한 번의 요청으로 가져와 데이터 전송을 최적화합니다. |
| **Stack Overflow 질문** | **Stack Exchange API**          | 사용자가 설정한 태그 기반의 안정적인 요청을 위해 **API 키**를 사용한 통합을 기본으로 합니다.             | 사용자 설정 기반의 고급 정렬 및 필터링(예: Hot, Unanswered)을 지원하여 위젯이 효과적인 인사이트 도구로 발전할 수 있도록 합니다.                                                                                                            |
| **기술 뉴스**           | RSS/뉴스 API (TBD)              | 간단한 비인증 API 키 또는 직접적인 RSS 피드 파싱을 활용합니다.                                           | 간단한 **HTTP GET** 요청 및 기본 데이터 파싱/정규화 작업이 필요합니다.                                                                                                                                                                     |

## 6. 비기능 요구 사항 (Non-Functional Requirements)

| 요구 사항 영역       | 구체적인 요구 사항                                                                                                                                                    | 구현 목표                                                                                             |
| :------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------- |
| **1. 성능**          | 모든 위젯 데이터는 초기 API 호출 성공 후 **300ms 이내**에 로드 및 표시되어야 합니다. 첫 페이지 로딩(FCP)은 **2초 이하**를 목표로 합니다.                              | 모든 로딩 위젯에 **스켈레톤 UI**를 구현합니다. React 훅과 코드 분할을 사용하여 렌더링을 최적화합니다. |
| **2. 반응형 UI**     | 주 설계 대상은 데스크톱(1280px+)이며, 태블릿(768px+) 환경에서는 **그리드 레이아웃이 재배치**되어야 합니다. 모바일 뷰는 MVP에서 **단순 1열 스크롤 모드**를 제공합니다. | `react-grid-layout`의 반응형 속성과 미디어 쿼리를 활용합니다.                                         |
| **3. 접근성 (A11Y)** | **키보드 기반 포커스 이동(Tab)을 지원**해야 합니다. 인터페이스는 텍스트 및 상호작용 요소에 대해 최소 **AA 명암 대비 기준(WCAG 2.1)**을 준수해야 합니다.               | 모든 대화형 요소에 명확한 포커스 상태와 적절한 ARIA 레이블을 보장합니다.                              |
| **4. 안정성**        | 시스템은 API 요청 실패 시 폴백(Fallback) UI를 구현해야 합니다. 외부 API의 **요청 한도(Rate Limit) 초과 시** 사용자에게 명확하고 구체적인 메시지를 제공해야 합니다.    | 각 위젯 컴포넌트 내에 독립적인 오류 상태를 구현합니다.                                                |

## 7. 에러 처리 및 재시도 전략 (Error Handling & Retry)

| 오류 시나리오                                      | 전략 및 동작                                                                                                                                                    | 기술적 세부 사항                                                                                                                                             |
| :------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **GitHub / Stack Overflow API 실패 (일시적 오류)** | 일시적 실패(예: 500/503 상태 코드)에 대해 **지수 백오프(Exponential Backoff)** 재시도 메커니즘을 구현합니다.                                                    | **1차 실패:** 2초 후 재시도. **2차 실패:** 4초 후 재시도. **3차 실패:** "데이터를 가져올 수 없습니다. 다시 시도해주세요." 오류 UI를 영구적으로 렌더링합니다. |
| **Rate Limit 초과 (403/429)**                      | 자동 재시도를 수행하지 않습니다. 영향을 받는 위젯 내에 영구적이고 명확한 **요청 한도 초과 UI**와 메시지를 표시합니다.                                           | API가 `Retry-After` 헤더 등을 제공하는 경우, 다음 재설정 시간을 사용자에게 안내해야 합니다.                                                                  |
| **위젯 단위 오류 UI**                              | 모든 위젯은 실패 시 명확한 오류 상태를 표시해야 하며, 오류 상태에는 수동으로 재시도 시퀀스를 시작할 수 있는 눈에 띄는 **"다시 시도" 버튼**이 포함되어야 합니다. | 오류는 위젯 경계 내로 제한되어 전체 대시보드가 오작동하는 것을 방지해야 합니다.                                                                              |

## 8. UI/UX 요구 사항 (UX 가이드라인)

| 요구 사항 영역             | 구체적인 가이드라인                                                                                                                                                  | 이유                                                                                                  |
| :------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------- |
| **1. 위젯 일관성**         | 모든 위젯은 **헤더 영역** (제목 및 설정 아이콘 위치)에 일관된 스타일을 유지해야 합니다. 너비와 높이 단위는 기본 **그리드 단위 시스템**을 기준으로 통일되어야 합니다. | 사용자가 위젯을 이동하거나 크기를 조정할 때 시각적 통일성과 예측 가능성을 보장합니다.                 |
| **2. 위젯 설정 패널**      | 각 위젯의 구성 항목(태그, 키워드, 저장소 등)은 전용 사이드 **Drawer 또는 Modal** 내에서 관리되어야 합니다.                                                           | 주 대시보드 보기를 복잡하게 만들지 않고 사용자 구성에 집중된 영역을 제공합니다.                       |
| **3. 로딩 상태**           | 모든 위젯은 데이터를 가져오는 동안 **스켈레톤 UI** (깜박이는 플레이스홀더)를 표시해야 합니다.                                                                        | 인지된 성능을 개선하고 시스템이 작동 중임을 시각적으로 피드백합니다.                                  |
| **4. 알림 (Notification)** | 주요 사용자 행동 또는 중요한 시스템 이벤트에 대해 간단한 토스트 알림 시스템을 구현합니다.                                                                            | 행동(예: "할 일 완료", "레이아웃 저장") 및 상태 변경(예: "API 호출 실패")에 대한 피드백을 제공합니다. |

## 9. 향후 확장 계획 (Future Enhancements)

| 확장 카테고리                   | 기능 설명                                                                                                                           | 가치 제안                                                        |
| :------------------------------ | :---------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------- |
| **1. 사용자 인증**              | **OAuth 기반 GitHub 로그인** (및 기타 서비스)을 구현하여 인증된 개인 데이터를 가져오고 데이터 정확도와 유용성을 높입니다.           | 프로젝트를 로컬 도구에서 교차 기기 애플리케이션으로 확장합니다.  |
| **2. 레이아웃 관리**            | 사용자가 여러 대시보드 레이아웃(예: "업무 레이아웃", "학습 레이아웃")을 **저장하고, 이름을 지정하고, 불러올 수 있도록** 허용합니다. | 개발자를 위한 개인화 및 컨텍스트 전환 기능을 크게 향상시킵니다.  |
| **3. 스타일링 및 커스터마이징** | **다크 모드** 토글 및 기본적인 테마 옵션을 구현합니다.                                                                              | 접근성을 개선하고 사용자 선호도를 충족시킵니다.                  |
| **4. AI/데이터 통찰**           | 일일 GitHub 활동 요약 또는 주요 뉴스 항목 요약과 같은 간단한 **AI 기반 요약 기능**을 통합합니다.                                    | 가공되지 않은 데이터 대신 고부가가치의 처리된 정보를 제공합니다. |
| **5. 플랫폼 확장성**            | 사용자가 새로운 위젯 유형을 동적으로 추가할 수 있는 **위젯 마켓플레이스/레지스트리** 구조를 위해 아키텍처를 설계합니다.             | 공동 기여 및 커뮤니티 주도 확장을 위한 프레임워크를 구축합니다.  |
